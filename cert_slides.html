전자서명법의 근본 문제
왜 한국만 사용자에게 개인키를 줬나
1999년 전자서명법의 핵심 논리:
한국 정부의 사고방식: "거래 당사자가 나중에 '내가 안 했다'고 부인하지 못하게 해야 한다" → "부인 방지에는 전자서명이 필요하다" → "전자서명에는 개인키가 필요하다" → "개인키를 사용자에게 줘야 한다" → 사용자가 USB/PC에 파일로 보관하고 직접 관리

한국 (1999년)

법이 특정 기술(PKI 인증서)을 유일한 전자서명 수단으로 지정
사용자가 개인키를 파일로 보관
분실, 유출, 갱신을 사용자가 책임

글로벌

"거래 부인 방지"는 서버측 로그 + 다중인증으로 해결
은행이 거래 기록을 보관하면 법적 효력 충분
사용자에게 개인키를 줄 이유가 없다
다른 나라는 "내가 안 했다"는 주장에 대해 서버 로그와 인증 기록으로 대응한다. 한국만 "사용자가 직접 전자서명을 했으니 부인할 수 없다"는 구조를 선택했다.

이론 vs 현실
개인키가 더 안전한 건 맞다. 하지만...

SSH에 쓰는 것과 같은 PKI 기술이다. 이론적으로 비밀번호보다 강력하다. 문제는 구현이다.

이론: 개인키의 장점

• 추측 불가능 (2048비트 랜덤)
• 피싱에 강함 (키 자체를 전송 안 함)
• 수학적으로 위조 불가능
• 부인 방지 (서명 사실을 부정 불가)

현실: 한국의 구현

• USB 메모리에 파일로 저장
• 파일 보호는 결국 비밀번호
• 파일 복사하면 누구나 사용 가능
• 개인키 + 비밀번호 = 비밀번호 수준

[한국의 선택] 비밀번호가 약하다 → 개인키가 강하다 → 사용자에게 개인키를 파일로 주자 → 그 파일을 비밀번호로 보호 → 결국 비밀번호 수준으로 회귀 [글로벌의 선택] 비밀번호가 약하다 → 비밀번호 + 다른 요소를 합치자 (MFA) → 비밀번호 + OTP, 생체인증, 하드웨어 키 → 나중에: 비밀번호 자체를 없애자 (FIDO2/패스키)
비유: "주차권이 비밀번호보다 안전하니까 국민에게 주차권을 나눠주자" → 주차권을 잃어버리니까 주차권에 비밀번호를 적어둠 → 결국 비밀번호가 뚫리면 주차권도 뚫림. 글로벌은 "번호판 인식으로 주차권 자체를 없애자"를 선택했다.

핵심 개념
다른 나라에는 사용자 개인키가 없다
전 세계 표준 방식에서 일반 사용자에게 개인키를 발급하지 않는다.
[한국 공인인증서 모델] 서버 개인키 → 서버가 관리 (HTTPS 암호화용) 사용자 개인키 → 사용자가 관리 ← 한국만 이렇게 한다 [글로벌 표준 모델] 서버 개인키 → 서버가 관리 (HTTPS 암호화용) 사용자 인증 → 비밀번호 + MFA (개인키 없음)
계층방식누가 관리
통신 암호화TLS (서버 인증서)서버
본인 확인ID/비밀번호서버가 해시 저장
추가 인증OTP, 생체인증서버 + 사용자 기기
거래 보호이상거래탐지(FDS)서버
부인 방지서버 로그 + 감사 추적서버
패스키(FIDO2)는 개인키를 사용하지만, 기기의 보안 칩에 저장되어 사용자가 존재 자체를 모른다. 지문이나 얼굴로 자동 사용. 한국 공인인증서와 같은 "개인키"지만 사용자 경험은 정반대다.

근본 진단
주차권을 자꾸 잃어버리니까 더 많은 주차권을 만들었다

주차권을 잃어버리는 문제를 "더 다양한 주차권"으로 해결하려 했다. 주차권을 없앨 생각은 안 했다.

한국의 해결책: "더 다양한 주차권을 만들자"

앱 주차권 (카카오 인증서)
핸드폰 주차권 (PASS 인증서)
은행 전용 주차권 (KB모바일 인증서)
정부 전용 주차권 (국민인증서)
결과: 주차권이 1개에서 7개로 늘었다

글로벌의 해결책: "주차권을 없애자"

번호판 인식 → 자동 입출차
주차권이 없으니 잃어버릴 것도 없음
복사 불가, 관리 불필요
프로그램 설치: 0개

주차권을 줬더니 파생 산업이 통째로 생겼다: 주차권 발급 (인증기관) → 주차권 보관함 (USB) → 주차권 관리 프로그램 (ActiveX) → 주차권 전달 프로그램 (Veraport) → 주차권 보호 프로그램 (TouchEn) → 주차권 분실 면책 (제9조) 주차권을 없애면 이 산업이 전부 사라진다. 그래서 아무도 바꾸고 싶어하지 않는다.
한국 주차장은 진작에 번호판 인식으로 바꿨다. 세계 금융도 바꿨다. 한국 은행만 아직도 주차권을 나눠주고 있다.

AI 시대
AI 시대에 주차권을 나눠주는 나라

AI 에이전트가 사용자를 대신해 일하는 시대가 왔다. 한국의 인증 체계는 AI가 작동할 수 없는 구조다.

왜 한국에는 KG이니시스가 필요한가

해외에서는 Stripe 하나로 모든 카드 결제가 된다. 한국에서는 PG사 → VAN사 → 카드사라는 다단계 구조가 필요하다.

왜 이런 구조가 생겼나: 한국 카드사들이 글로벌 표준 프로토콜(Visa/Mastercard 직접 연동)을 안 쓴다. 카드사마다 통신 방식이 달라서, 가맹점이 일일이 연동할 수 없다. 그래서 "중간에서 연결해줄게" → PG사(이니시스, KCP), "거래를 중계해줄게" → VAN사(NICE, KSNET)가 탄생했다. Stripe가 한국에 못 들어오는 이유이기도 하다.

1990년대 세계
인터넷 보안에 국제 표준이 생기다

1994년, 넷스케이프(Netscape)라는 회사가 웹 브라우저를 만들었다. 사람들이 처음으로 인터넷에서 물건을 사기 시작했다. 그런데 문제가 있었다 — 카드 번호를 입력하면 누군가 중간에서 훔쳐볼 수 있었다.

그래서 넷스케이프가 SSL이라는 기술을 만들었다. SSL은 인터넷 통신을 암호화하는 기술이다. 쉽게 말하면, 편지를 봉투에 넣어서 보내는 것이다. 중간에 누가 가로채도 봉투를 열 수 없다. 웹 브라우저 주소창에 자물쇠 모양이 뜨면 SSL(지금은 TLS로 발전)이 작동하고 있다는 뜻이다.

연도사건의미
1994넷스케이프가 SSL 1.0 개발인터넷 암호화의 시작 (보안 결함으로 비공개)
1995SSL 2.0 공개최초의 공개 암호화 통신. 아마존, 이베이가 전자상거래 시작
1999TLS 1.0 발표SSL의 후속 표준. 국제 표준으로 채택 (RFC 2246)
핵심: 보안을 브라우저와 서버 사이에서 처리했다. 사용자 PC에 별도 프로그램을 설치할 필요가 없었다. 이것이 이후 25년간 세계 보안의 기본 원칙이 된다.
