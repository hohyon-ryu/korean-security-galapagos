4가지 구조적 문제와 해결의 실마리

Section 1

4대 구조적 문제

각각의 문제가 왜 생겼고, 왜 25년간 고착됐는지 추적한다.

결제 보안의 갈라파고스

  • 1999년 미국 암호화 수출 금지 → 한국이 독자 암호화(SEED) + ActiveX로 해결 → 1년 뒤 미국이 수출 제한을 풀었지만, 한국은 이미 ActiveX 위에 인프라를 올려놓은 뒤였다
  • 전자서명법이 공인인증서만 법적 효력으로 인정 → 21년간 Internet Explorer 종속
  • 2005년 인터넷뱅킹 해킹 이후 보이스피싱 대응 명목으로 PC 보안프로그램 의무화 → 실제 보이스피싱(전화 사기)에는 무력
  • 보안프로그램의 실제 기능: 은행의 면책 도구. "설치하라고 했잖아" → 사고 책임이 사용자에게 전가
  • 규제가 만든 이권 구조: 보안업체 매출을 규제가 보장. 업체는 규제 유지 로비, 퇴직 공무원은 보안업체 취업(회전문 인사)

망분리의 덫

  • 2006년부터 공무원 업무 PC에서 인터넷 완전 차단 — 해킹 방지가 목적
  • 결과: 온나라시스템(정부 행정 시스템)이 2022년부터 3년간 해킹당했지만 발견하지 못함. 망분리는 침입 탐지 능력을 제거한다
  • 미국은 2022년 바이든 행정명령(EO 14028)으로 연방정부 제로트러스트 전환 시작
  • 영국 공무원은 Google Workspace를 업무용으로 사용. 클라우드 기반, AI 도구 활용 가능
  • 외교부가 "에이전틱 AI" 사업을 추진 중이지만, 망분리 환경에서 AI 에이전트는 외부 정보에 접근할 수 없다 — 구조적 모순

HWP 문서 독점

  • 대한민국 공문서가 한컴오피스 독점 포맷(.hwp/.hwpx)에 갇혀 있음
  • AI가 HWP를 제대로 읽을 수 없어 행정 문서 자동화, AI 분석, 검색 모두 불가
  • 연간 수십억 원의 라이선스 비용이 단일 기업에 귀속
  • 시민이 공문서를 열려면 6만 원짜리 프로그램을 구매하거나 불완전한 뷰어 사용
  • 미국, 영국, EU는 PDF 또는 웹 문서(HTML)를 표준으로 사용 — 누구나, 어떤 기기에서나 열 수 있음

보안프로그램 과잉

  • 공무원 PC에 10종 이상의 보안프로그램 설치 의무. 부팅에 수 분, 업무 속도 50% 저하
  • KAIST/USENIX Security 2025 연구: 한국 보안프로그램 7종을 분석해 심각한 취약점 19건 발견
  • 북한 라자루스 그룹이 Veraport(2020년), MagicLine4NX(2023년)를 해킹 통로로 사용 — 보안프로그램이 공격 벡터가 된 사례
  • TouchEn nxKey(2023년): 전 국민이 설치한 키보드 보안 프로그램에서 키로거 수준의 취약점 발견 — 모든 키 입력을 엿볼 수 있는 구조
Section 2

25년간의 타임라인

시작은 합리적 이유였다. 문제는 세계가 전환할 때 한국만 남았다는 것이다.

1999
미국 암호화 수출 금지 → 한국 독자 암호화(SEED) + ActiveX 채택. 전자서명법 제정 → 공인인증서만 법적 효력.
2000
미국 수출 제한 해제. 한국은 이미 ActiveX 위에 전체 금융 인프라를 구축한 뒤. 되돌리기엔 늦었다.
2005
한국 최초 인터넷뱅킹 해킹 사고 → 보안프로그램 설치 의무화 시작. 이후 20년간 고착.
2006
망분리 정책 도입. 공무원 업무 PC에서 인터넷 완전 차단.
2008
Google Chrome 출시 (샌드박스 보안). 세계는 브라우저 격리 기술로 전환. 한국은 반대 방향으로 달렸다.
2020
북한 라자루스, Veraport 공급망 공격으로 악성코드 배포. 같은 해 공인인증서 21년 만에 폐지.
2022
미국 바이든 행정명령(EO 14028) — 연방정부 제로트러스트 전환. 같은 시기 온나라시스템 해킹 시작(3년간 미발견).
2023
TouchEn nxKey 전 국민 키로거 취약점 공개. 북한 MagicLine4NX로 기업 침입. 영국-한국 공동 경보 발령.
2024
국정원 N2SF(국가망보안체계) 발표 — 제로트러스트 방향 제시. 실제 전환 속도는 매우 더딤.
2026
외교부 AI 2단계 사업 착수. 한국 시민은 여전히 프로그램 5개를 깔고 있다.
Section 3

세계는 어떻게 하나

같은 문제를 다른 나라는 이미 해결했다.

Section 4

에이전트 시대의 보안 구조

AI 에이전트가 사람을 대신해 금융, 행정, 업무를 처리하는 시대. 현재 보안 체계는 이 구조와 호환되지 않는다.

!

현재 구조의 근본 문제

현재 보안은 "사람이 브라우저에서 직접 조작"하는 것을 전제한다. 프로그램 설치, 키보드 보안, 화면 캡처 방지 — 전부 사람의 눈과 손을 가정한 설계. AI 에이전트는 브라우저를 쓰지 않고, 키보드를 치지 않으며, 화면을 보지 않는다.

1

API-first 인증

에이전트는 브라우저가 아니라 API로 통신한다. OAuth 2.0 스코프 기반 권한 위임이 필요하다. "이 에이전트는 잔액 조회만 가능, 이체는 불가" 같은 세분화된 권한 설정.

2

에이전트 신원 확인

사람이 FIDO2(지문/얼굴)로 인증 → 에이전트에 시간제한 토큰 발급. 에이전트는 30분간만 특정 작업을 수행할 수 있고, 시간이 지나면 재인증이 필요.

3

행위 기반 보안

"누구인가"가 아니라 "무엇을 하는가"로 판단. 평소 패턴과 다른 행위(갑자기 대량 이체, 새벽 3시 접속)를 실시간 탐지. 이상 행위 시 즉시 차단.

4

감사 추적 (Audit Trail)

에이전트의 모든 행위를 서버에서 로깅. 누가, 어떤 에이전트에, 어떤 권한을, 언제 부여했고, 에이전트가 무엇을 했는지 전부 기록. 사후 추적과 책임 소재가 명확해진다.

Section 5

무엇을 요구해야 하나

행동가로서 각 기관에 요구할 구체적 사항.

국회

  • → 전자금융거래법 개정: "뭘 깔아라" → "뭘 달성해라" (목표 기반 규제 전환)
  • → 사고 책임을 금융기관으로 전환 (EU PSD2 모델)
  • → 망분리 규정 개정: 제로트러스트 허용 법적 근거 마련
  • → 행정업무운영규정 개정: HWP 독점 해소

금감원

  • → 보안프로그램 설치 의무 규정 폐지
  • → PCI DSS, 3D Secure 등 글로벌 보안 표준 채택
  • → 금융기관 서버 보안 투자 의무화
  • → 보안프로그램 HTTPS 중간 해독(MITM) 금지

은행

  • → 서버측 이상거래탐지(FDS) 고도화
  • → FIDO2/패스키 도입
  • → 클라이언트 프로그램 의존도 단계적 제거
  • → 사고 발생 시 기관 책임 원칙 수용

행안부

  • → 공문서 개방형 포맷(PDF/ODF) 의무화
  • → N2SF 제로트러스트 전환 가속화
  • → 공무원 AI 활용 가이드라인 수립
  • → 보안프로그램 10종 → 5종 이내 축소

시민

  • → 국회 청원: 전자금융거래법, 망분리 규정 개정 요구
  • → 보안프로그램 강제 설치에 문제 제기
  • → 이 보고서와 데이터를 공유, 확산
  • → 금융 사고 시 기관 책임 요구
