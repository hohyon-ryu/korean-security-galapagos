AI 시대 대한민국 IT 인프라 전환 실행 계획

Section 1

결제 보안 전환 로드맵

금융 IT 보안 체계를 클라이언트 프로그램 의존에서 서버 기반 국제 표준으로 전환하는 3단계 계획

Phase 1 — 6개월 (즉시 실행)
  • 금감원: 보안프로그램 설치 의무 규정 유예 고시 발령
  • 은행: FIDO2/패스키 시범 도입 (카카오뱅크, 토스뱅크 선행)
  • 금융위: PCI DSS 준수 시 클라이언트 프로그램 면제 제도 도입
Phase 2 — 1년 (법적 기반 마련)
  • 전자금융거래법 개정: 목표 기반 규제로 전환 — "키보드 보안 프로그램 설치" → "키 입력 보호 달성"
  • 사고 책임 전환: EU PSD2 모델 도입 (금융기관 책임 원칙)
  • 보안프로그램의 HTTPS 중간 해독 금지 규정 신설
Phase 3 — 2년 (전면 전환 완료)
  • 전 금융기관 서버측 보안 전환 완료
  • 토큰화(Tokenization) 의무화 — 카드번호 직접 전송 금지
  • 글로벌 결제사(Stripe 등) 진입 허용을 위한 전자금융거래법 추가 개정
Section 2

망분리 → 제로트러스트 전환 로드맵

모든 업무를 동일 수준으로 차단하는 방식에서, 데이터 등급에 따라 차등 보호하는 체계로 전환

핵심 원칙: 3등급 분류

C등급 — 일반 행정 (80%)

제로트러스트로 전환. 인터넷 + AI 도구 완전 허용. 클라우드 기반 업무 환경.

S등급 — 민감 업무 (15%)

제한적 인터넷. 승인된 AI 도구만 허용. 데이터 분류에 따른 접근 제어.

TS등급 — 국가 기밀 (5%)

물리적 분리 유지. 전용 네트워크. 기존 방식 그대로.

외교부, 국방부 등 고보안 기관

  • 현재: 모든 업무를 TS 수준으로 취급 → 100% 차단
  • 전환 후: 실제 기밀 5%만 물리적 분리, 나머지 95%는 AI 활용 가능
  • 외교부 AI 2단계 사업(에이전틱 AI)이 성공하려면 제로트러스트 전환 필수
Phase 1 — 6개월 (시범 전환)
  • 국정원: N2SF 가이드라인 실제 적용 가속화
  • 행안부: 일반 행정부서 시범 전환 (서울시, 세종시)
  • 범정부: 데이터 등급 분류 기준 수립
Phase 2 — 1년 (중앙부처 전환)
  • 중앙부처 일반 행정(C등급) 전면 전환
  • AI 도구 업무 활용 가이드라인 수립
  • 모바일 공무원증 기반 다중인증(FIDO2) 도입
Phase 3 — 2년 (전면 확산)
  • 지방자치단체 전면 전환
  • 외교부, 국방부 S등급 업무 제한적 전환
  • 온나라시스템 GPKI → FIDO2 인증 전환 완료
Section 3

공문서 포맷 전환 로드맵

HWP 단일 포맷 의존을 해소하고, AI가 읽고 분석할 수 있는 개방형 포맷으로 전환

Phase 1 — 6개월 (제도 기반)
  • 행안부: 행정업무운영규정 개정 — Markdown/PDF 허용
  • 시민 배포 문서: PDF/HTML 의무, HWP only 금지
Phase 2 — 1년 (시스템 전환)
  • 온나라 문서시스템에 Markdown 기반 에디터 지원
  • 기존 HWP 문서 → PDF 일괄 변환 (AI 활용)
Phase 3 — 2년 (완전 전환)
  • HWP 신규 작성 완전 중단
  • 수십 년간 축적된 공공문서 AI 검색, 분석 가능
Section 4

행정 보안프로그램 정상화

실효성 없는 클라이언트 보안프로그램 남용을 종식하고, 서버사이드 보안 체계로 전환

Phase 1 — 6개월 (즉시 축소)
  • 행안부: 보안프로그램 10종 → 5종 이내 축소 지침
  • 독립적 보안 감사 체계 도입
Phase 2 — 1년 (체계 전환)
  • 클라이언트 보안 → 서버사이드 보안 전환
  • 보안프로그램 실효성 검증 의무화
Phase 3 — 2년 (최소화 완료)
  • 제로트러스트 기반 보안 체계 완성
  • 클라이언트 프로그램 최소화 (백신 1종 수준)
Section 5

에이전트 시대 보안 및 접근 구조

AI 에이전트가 사람을 대리하여 업무를 수행하는 시대에 맞는 보안 아키텍처

현재 구조의 한계

  • 사람이 브라우저에서 직접 조작하는 것을 전제로 설계됨
  • "프로그램 설치 → 클릭 → 인증서 비밀번호" = AI 에이전트가 수행 불가
  • 망분리 = 에이전트가 외부 정보 수집, 연동 자체 불가

에이전트 시대 보안 아키텍처 (4 Layer)

Layer 1

신원 확인 (Identity)

  • 사람: FIDO2 패스키 (지문/얼굴)
  • 에이전트: OAuth 2.0 + 위임 토큰 (사람이 FIDO2로 인증 → 에이전트에 시간제한 권한 부여)
  • 디바이스: 기기 인증서 (TPM/Secure Enclave)
Layer 2

권한 관리 (Authorization)

  • 스코프 기반: 에이전트마다 할 수 있는 일의 범위를 명시 (예: "잔액 조회만 가능", "100만 원 이하 이체만 가능")
  • 시간 제한: 토큰 유효기간 설정 (예: 1시간)
  • 컨텍스트 인식: 시간, 위치, 디바이스에 따른 동적 권한 조정
Layer 3

행위 감시 (Behavior Monitoring)

  • "누구인가"가 아니라 "무엇을 하는가"로 판단
  • 이상거래탐지(FDS) 확장: 에이전트의 행위 패턴 모니터링
  • 실시간 이상행위 차단 + 사후 감사 추적(Audit Trail)
Layer 4

API 기반 서비스 (API-first)

  • 은행, 공공 서비스: 웹 UI 뿐 아니라 API 엔드포인트 제공
  • 에이전트가 프로그래밍 방식으로 서비스에 접근 가능
  • 표준 프로토콜: OpenAPI, OAuth 2.0, WebAuthn

법률 개정 필요 사항

IT 인프라 전환을 위해 개정이 필요한 6개 법률 및 규정

법률 현재 개정 방향 담당
전자금융거래법 특정 기술(프로그램) 지정 목표 기반 규제 금융위
전자서명법 폐지됐지만 관행 잔존 다양한 인증 수단 허용 명시 과기부
국가정보화기본법 망분리 근거 제로트러스트 허용 명시 행안부
행정업무운영규정 HWP 사실상 강제 개방형 포맷 의무 행안부
전자금융감독규정 보안프로그램 설치 의무 서버 보안 기준으로 전환 금감원
개인정보보호법 클라이언트 수집 허용 최소 수집 원칙 강화 개인정보위
